Complete Guide to Distributed Training

From Zero to Hero

Part 1: Why Distributed Training Exists

The Fundamental Problem

Imagine you're training GPT-4. It has 1.76 trillion parameters. Each parameter is a number (usually stored as a 32-bit float = 4 bytes).

Just storing the model weights:

1.76 trillion × 4 bytes = 7 terabytes

But training needs more than just weights:

Weights:          7 TB
Gradients:        7 TB
Optimizer states: 14 TB  (Adam keeps 2 copies)
Total:           ~28 TB of memory

Problem: A single H100 GPU has 80 GB of memory. You'd need ~350 GPUs just to hold everything in memory, let alone actually train!

The Three Goals of Distribution

  • Model Parallelism: Split the model across GPUs because it's too big for one
  • Data Parallelism: Train on more data simultaneously to speed up training
  • Pipeline Parallelism: Keep all GPUs busy all the time (no idle waiting)

Part 2: Data Parallelism (The Simplest Approach)

Basic Concept

Analogy: Imagine 4 students grading the same essay. Each student:

  1. Gets a copy of the rubric (model weights)
  2. Grades different parts of the essay (different data batches)
  3. Compares notes and updates the rubric together (synchronize gradients)
  4. Repeats with next essay

How It Works Step-by-Step

Setup (before training starts):

GPU 0: Copy of full model (7 TB)
GPU 1: Copy of full model (7 TB)
GPU 2: Copy of full model (7 TB)
GPU 3: Copy of full model (7 TB)

Training iteration:

1. Forward Pass (each GPU independently):
   GPU 0: Processes batch of 32 images (cats)
   GPU 1: Processes batch of 32 images (dogs)
   GPU 2: Processes batch of 32 images (birds)
   GPU 3: Processes batch of 32 images (fish)

2. Backward Pass (each GPU independently):
   GPU 0: Computes gradients from its batch
   GPU 1: Computes gradients from its batch
   GPU 2: Computes gradients from its batch
   GPU 3: Computes gradients from its batch

3. Synchronization (GPUs communicate):
   All GPUs share their gradients and compute the average
   This is called "All-Reduce"

4. Update (each GPU independently):
   Each GPU updates its copy of the model with averaged gradients
   All models are now identical again

The All-Reduce Operation

What is it? A communication pattern where every GPU needs to know the sum (or average) of values from all GPUs.

GPU 0 computed: gradient = 5
GPU 1 computed: gradient = 3
GPU 2 computed: gradient = 7
GPU 3 computed: gradient = 1

After All-Reduce:
All GPUs have: average gradient = (5+3+7+1)/4 = 4

Naive approach (slow):

GPU 0 sends to GPU 1, GPU 2, GPU 3
GPU 1 sends to GPU 0, GPU 2, GPU 3
...
Total messages: 12
Time complexity: O(N) where N = number of GPUs

Ring All-Reduce (smart, what everyone uses):

Arrange GPUs in a ring: 0 → 1 → 2 → 3 → 0

Phase 1 (Reduce-Scatter): Pass partial sums around ring
Phase 2 (All-Gather):     Pass final results around ring

Total messages: 2(N-1)
Time complexity: O(1) regardless of number of GPUs!

Data Parallelism Variants

Synchronous Data Parallelism (most common)

All GPUs wait for each other after each batch. Guarantees all models stay in sync. Like students waiting for everyone to finish grading before discussing.

Asynchronous Data Parallelism

GPUs update a central parameter server independently. Faster but can be unstable (some gradients might be stale). Like students submitting rubric updates without waiting for others.

Gradient Accumulation

Run multiple forward/backward passes before synchronizing. Simulates larger batch sizes without more memory.

for micro_batch in [batch1, batch2, batch3, batch4]:
    loss = forward(micro_batch)
    gradients += backward(loss)  # accumulate, don't update yet

# Now synchronize and update once
all_reduce(gradients)
update_weights(gradients)

Part 3: Model Parallelism (When Model Doesn't Fit)

The Problem Data Parallelism Doesn't Solve

With data parallelism, every GPU needs a full copy of the model. If the model is 28 TB and you have 80 GB GPUs, you still can't train!

Tensor Parallelism (Horizontal Splitting)

Analogy: Imagine a giant Excel spreadsheet that's too big for one computer screen. You split it vertically so each screen shows different columns, but all screens show the same rows simultaneously.

How it works for a matrix multiplication:

Normal computation (single GPU):
# Input: X with shape [batch_size, input_dim]
# Weight: W with shape [input_dim, output_dim]
# Output: Y = X @ W
Y = X @ W

Tensor parallelism (2 GPUs):
# Split weight matrix W vertically:
# GPU 0 gets W[:, :output_dim/2]  (left half)
# GPU 1 gets W[:, output_dim/2:]  (right half)

# Both GPUs get full input X
GPU_0: Y_left  = X @ W_left   # produces left half of output
GPU_1: Y_right = X @ W_right  # produces right half of output

# Final output is concatenated: Y = [Y_left | Y_right]

Real Example — Transformer MLP Layer:

Original (single GPU):
X [batch=4, seq=512, dim=4096]
    ↓
W1 [4096, 16384]  ← 67M parameters
    ↓
Activation [batch=4, seq=512, dim=16384]
    ↓
W2 [16384, 4096]  ← 67M parameters
    ↓
Output [batch=4, seq=512, dim=4096]

Tensor Parallel (2 GPUs):
GPU 0 gets W1_left [4096, 8192] and W2_top [8192, 4096]
GPU 1 gets W1_right [4096, 8192] and W2_bottom [8192, 4096]

Communication needed:
- After W1: Concatenate left and right halves
- After W2: Sum contributions from both GPUs

Attention Mechanism with Tensor Parallelism:

Original attention (single GPU):
Q, K, V = X @ W_Q, X @ W_K, X @ W_V  # each W is [4096, 4096]
Attention = softmax(Q @ K^T / √d) @ V

With 4 GPUs (splitting attention heads):
GPU 0: Handles heads 0-7    (8 heads)
GPU 1: Handles heads 8-15   (8 heads)
GPU 2: Handles heads 16-23  (8 heads)
GPU 3: Handles heads 24-31  (8 heads)

Each GPU:
- Stores 1/4 of W_Q, W_K, W_V weights
- Computes attention for its heads independently
- At the end, all-gather to combine results

Pipeline Parallelism (Vertical Splitting)

Analogy: An assembly line for building cars. Station 1 builds the chassis, Station 2 adds the engine, Station 3 adds the body, Station 4 adds the interior. Each station (GPU) specializes in one part.

Simple Example — 4 Layer Model on 4 GPUs:
GPU 0: Layer 1 (embedding + first transformer block)
GPU 1: Layer 2 (second transformer block)
GPU 2: Layer 3 (third transformer block)
GPU 3: Layer 4 (output layer + loss)

Timeline without pipelining (inefficient):
Time →
GPU 0: [Process batch 1] [idle] [idle] [idle] [Process batch 2] ...
GPU 1: [idle] [Process batch 1] [idle] [idle] [idle] ...
GPU 2: [idle] [idle] [Process batch 1] [idle] [idle] ...
GPU 3: [idle] [idle] [idle] [Process batch 1] [idle] ...

Problem: 75% of time, GPUs are idle!

GPipe (Google's Pipeline Parallelism)

The Solution: Process multiple mini-batches simultaneously, like a real assembly line.

Split each batch into 4 micro-batches

Timeline with pipelining:
Time →     mb1    mb2    mb3    mb4    mb1    mb2    mb3    mb4
GPU 0:     [F1]   [F2]   [F3]   [F4]   [B1]   [B2]   [B3]   [B4]
GPU 1:     [idle] [F1]   [F2]   [F3]   [F4]   [B1]   [B2]   [B3]
GPU 2:     [idle] [idle] [F1]   [F2]   [F3]   [F4]   [B1]   [B2]
GPU 3:     [idle] [idle] [idle] [F1]   [F2]   [F3]   [F4]   [B1]

F = Forward pass, B = Backward pass, mb = micro-batch

After warmup, all GPUs are busy!
Efficiency = 4/(4+3) = 57%  (with 4 micro-batches)
With 8 micro-batches:  8/(8+3)  = 73%
With 16 micro-batches: 16/(16+3) = 84%
Memory Trade-off: More micro-batches = better GPU utilization, but need to store activations for each micro-batch. This can use a lot of memory!

PipeDream (Microsoft's Improvement)

The Innovation: Don't wait for all forward passes to finish before starting backward passes.

Timeline:
Time →
GPU 0:  1F  2F  3F  4F  1B  2B  3B  4B  5F  6F  ...
GPU 1:  --  1F  2F  3F  4F  1B  2B  3B  4B  5F  ...
GPU 2:  --  --  1F  2F  3F  4F  1B  2B  3B  4B  ...
GPU 3:  --  --  --  1F  2F  3F  4F  1B  2B  3B  ...

Reaches steady state faster!

The Weight Version Problem:

Micro-batch 1 forward uses weight version W_0
By the time micro-batch 1 backward runs, weights updated to W_3
Gradient is "stale" — computed for old weights!

Solution: Keep multiple weight versions in memory
Trade memory for correctness

Part 4: Advanced Parallelism Strategies

3D Parallelism (Megatron-LM)

The Idea: Combine all three types of parallelism!

Example Configuration for GPT-175B on 1024 GPUs:
Total GPUs: 1024
Arranged as: 8 (pipeline) × 8 (tensor) × 16 (data)

Within each slice (64 GPUs = 8 pipeline × 8 tensor):
┌─────────────────────────────────&boxgr;
│  Layer 1:     [GPU 0-7]  (tensor)  │  ← Pipeline stage 0
├─────────────────────────────────&boxgr;
│  Layer 2-3:   [GPU 8-15]           │  ← Pipeline stage 1
├─────────────────────────────────&boxgr;
│  ...                               │
├─────────────────────────────────&boxgr;
│  Layer 14-15: [GPU 56-63]          │  ← Pipeline stage 7
┐─────────────────────────────────&boxgl;

Each layer's weights split across 8 GPUs (tensor parallel)
16 identical slices running data parallelism

Communication patterns:

  1. Within tensor parallel group (8 GPUs): Very frequent, needs fast interconnect (NVLink)
  2. Between pipeline stages: Less frequent, sends activations/gradients
  3. Across data parallel replicas (16 groups): Only at gradient sync time

ZeRO (Zero Redundancy Optimizer) — DeepSpeed

The Key Insight: In data parallelism, we replicate everything. But do we need to?

Traditional Data Parallelism Waste:
4 GPUs, each stores:
  Model weights:    7 TB
  Gradients:        7 TB
  Optimizer states: 14 TB
  Total per GPU:    28 TB
  Total across 4 GPUs: 112 TB

But the gradients and optimizer states are identical after all-reduce!
We're storing the same thing 4 times!

ZeRO Stage 1: Partition Optimizer States

Instead of each GPU storing all optimizer states:

GPU 0: Stores optimizer states for parameters 0-25%
GPU 1: Stores optimizer states for parameters 25-50%
GPU 2: Stores optimizer states for parameters 50-75%
GPU 3: Stores optimizer states for parameters 75-100%

When updating weights:
  GPU 0 updates its 25% and broadcasts to others
  GPU 1 updates its 25% and broadcasts to others
  ...

Memory saved: 14 TB → 3.5 TB per GPU

ZeRO Stage 2: Also Partition Gradients

GPU 0: Stores gradients for parameters 0-25%
GPU 1: Stores gradients for parameters 25-50%
...

During backward pass:
  Compute all gradients locally
  Reduce-scatter: each GPU gets average of its partition
  Discard the rest

Memory saved: (14 + 7) TB → (3.5 + 1.75) TB per GPU

ZeRO Stage 3: Also Partition Model Weights

GPU 0: Stores weights for parameters 0-25%
GPU 1: Stores weights for parameters 25-50%
...

During forward pass:
  Before computing layer L: all-gather weights for layer L
  Compute layer L
  Discard weights for layer L
  Repeat for next layer

Memory saved: 28 TB → 7 TB per GPU (4x reduction!)
Extra cost: More communication (all-gather weights repeatedly)

ZeRO-Infinity: Offload to CPU and NVMe

GPU memory: Only what's actively being computed
CPU memory: Recently used weights, optimizer states
NVMe SSD:  Everything else

Example — Training GPT-175B on a single GPU:
  GPU (32 GB):    Current layer being computed
  CPU RAM (512 GB): Next few layers + optimizer states
  SSD (2 TB):    Rest of model

Achieves: Training trillion parameter models on consumer hardware!
Trade-off: Much slower, but makes impossible → possible

Part 5: Communication Primitives Deep Dive

All distributed training relies on GPUs talking to each other. Here are the fundamental operations:

Broadcast

One GPU sends data to all others

GPU 0: Has value X = 42    →  All GPUs have: X = 42
GPU 1: Wants value X
GPU 2: Wants value X
GPU 3: Wants value X

Used for: Sending initial weights, synchronizing randomness

Reduce

Combine values from all GPUs to one GPU

GPU 0: X = 5  ────&boxgr;
GPU 1: X = 3  ────&boxgr;  Reduce (sum) to GPU 0:
GPU 2: X = 7  ────&boxgr;  GPU 0: X = 16
GPU 3: X = 1  ────&boxgl;

Used for: Collecting losses, aggregating metrics

All-Reduce

Reduce + Broadcast = Everyone gets the combined result

Before:                    After all-reduce (sum):
GPU 0: X = 5               GPU 0: X = 16
GPU 1: X = 3               GPU 1: X = 16
GPU 2: X = 7               GPU 2: X = 16
GPU 3: X = 1               GPU 3: X = 16

Used for: Gradient synchronization (most important!)

Scatter

Split data from one GPU to all GPUs

GPU 0: [A, B, C, D]  →  GPU 0: [A]
                          GPU 1: [B]
                          GPU 2: [C]
                          GPU 3: [D]

Used for: Distributing data batches

Gather

Collect data from all GPUs to one GPU

GPU 0: [A]  →  GPU 0: [A, B, C, D]
GPU 1: [B]
GPU 2: [C]
GPU 3: [D]

Used for: Collecting predictions, logging

All-Gather

Gather + Broadcast = Everyone gets all data

Before:          After all-gather:
GPU 0: [A]       GPU 0: [A, B, C, D]
GPU 1: [B]       GPU 1: [A, B, C, D]
GPU 2: [C]       GPU 2: [A, B, C, D]
GPU 3: [D]       GPU 3: [A, B, C, D]

Used for: ZeRO Stage 3 weight gathering, tensor parallelism

Reduce-Scatter

Reduce + Scatter = Each GPU gets a piece of the sum

Before:                       After reduce-scatter (sum):
GPU 0: [5, 3, 7, 1]          GPU 0: [8]   (5+2+1+0)
GPU 1: [2, 4, 6, 8]          GPU 1: [10]  (3+4+1+2)
GPU 2: [1, 1, 1, 1]          GPU 2: [18]  (7+6+1+4)
GPU 3: [0, 2, 4, 6]          GPU 3: [15]  (1+8+1+6)

Used for: ZeRO optimizer state updates, efficient gradient sync

Part 6: Network Topologies and Hardware

Why Network Matters

Training a layer:
  Compute time:        10ms  (matrix multiply on GPU)
  Communication time:  50ms  (send activations to next GPU)

If communication is 5x slower than compute,
your GPUs sit idle 83% of the time!

GPU Interconnects

InterconnectBandwidthUse CaseLatency
PCIe~16 GB/sGPU ↔ CPU~microseconds
NVLink 4.0~600 GB/sGPU ↔ GPU (within node)~nanoseconds
InfiniBand HDR~200 GB/sNode ↔ Node~1 μs

NVLink is 37x faster than PCIe!

Network Topologies

Fully Connected (ideal but expensive)

Every GPU directly connected to every other.
Works great up to ~8 GPUs (single server with NVLink).
Beyond that: too expensive, physically impossible.

Ring Topology (most common)

GPU 0 → GPU 1 → GPU 2 → GPU 3 → GPU 0

Advantages:  Simple, cheap. All-reduce in constant time. Each GPU only needs 2 connections.
Disadvantages: Single point of failure. Not great for non-all-reduce operations.

Tree Topology

          GPU 0
         ⁄     \
    GPU 1       GPU 2
   ⁄    \     ⁄    \
 GPU 3  GPU 4 GPU 5  GPU 6

Advantages:  Great for broadcast/reduce. Logarithmic depth.
Disadvantages: Root can be bottleneck. All-reduce slower than ring.

Torus/Mesh (high-end clusters)

2D Torus (edges wrap around):
GPU 0 — GPU 1 — GPU 2 — GPU 3
  |       |       |       |
GPU 4 — GPU 5 — GPU 6 — GPU 7
  |       |       |       |
GPU 8 — GPU 9 — GPU10 — GPU11

Advantages:  Multiple paths, resilient, great bisection bandwidth.
Disadvantages: Complex, expensive, harder to program.

Part 7: Fault Tolerance and Checkpointing

Why Failures Matter

Training GPT-4:
  25,000 GPUs for 100 days
  If each GPU has 0.1% daily failure rate:
    Expected failures per day: 25,000 × 0.001 = 25 failures/day
    Will definitely experience failures!

Checkpointing Strategies

Naive Checkpointing

for epoch in range(100):
    for batch in dataloader:
        train_step(batch)

    save_checkpoint({
        'model': model.state_dict(),
        'optimizer': optimizer.state_dict(),
        'epoch': epoch
    })

Problem: If training fails mid-epoch, lose all progress since last checkpoint
Size: Full model state might be 28 TB!
Time: Saving 28 TB takes ~5 minutes on fast storage

Incremental Checkpointing

checkpoint_every_n_steps = 100

for step in range(total_steps):
    train_step()
    if step % checkpoint_every_n_steps == 0:
        save_checkpoint(step)

Advantages:    Lose at most 100 steps of work
Disadvantages: More frequent I/O, potential slowdown

Asynchronous Checkpointing

Main training thread:
  Step 1000: Train → Trigger checkpoint (non-blocking)
  Step 1001: Train...
  Step 1002: Train...

Background thread:
  Copying model state to CPU...
  Writing to disk...
  Done!

GPUs keep training while checkpoint happens.
Needs extra CPU RAM to hold copy.

Distributed Checkpointing (for large models)

With ZeRO Stage 3, each GPU only has part of the model.

Option 1 (Centralized): Gather everything to one node, save.
  Problem: Aggregator needs 28 TB RAM, slow.

Option 2 (Distributed): Each GPU saves its partition.
  GPU 0 saves partition_0.pt (7 TB)
  GPU 1 saves partition_1.pt (7 TB)
  ...
  Faster (parallel I/O) but more complex to manage.

Elastic Training

Start training with 100 GPUs
GPU 47 fails → System detects failure
Checkpoint current state → Restart with 99 GPUs
Redistribute work → Continue from checkpoint

Can even add GPUs dynamically if more become available!

Gradient Checkpointing (different thing!)

Note: This is about saving memory, not saving to disk. 
Normal training:
  Forward: Store all activations (needed for backward)
  Memory: O(number of layers)

Gradient checkpointing:
  Forward: Only store checkpoints every N layers
  Backward: Recompute missing activations on-the-fly
  Memory: O(√(number of layers))

Example with 12 layers:
  Normal:              Store activations for L1, L2, ..., L12 (12 items)
  Checkpoint every 4:  Store L4, L8, L12 (3 items)

  When computing gradients for L2:
    Re-run forward from checkpoint L4 → L1, L2
    Use those activations for backward, then discard

Trade-off: 33% more computation, but saves tons of memory!

Part 8: Practical Considerations

Batch Size and Learning Rate

Scaling Rule:
Original:    1 GPU, batch size 32, learning rate 0.001
With 4 GPUs: effective batch size = 4 × 32 = 128

Need to scale learning rate!
Common rule: LR scales linearly with batch size
New learning rate: 0.001 × 4 = 0.004

Why? Larger batches → less noisy gradients → can take bigger steps
Warmup:
# Don't immediately use large learning rate
# Gradually increase from 0 to target

for step in range(warmup_steps):
    lr = target_lr * (step / warmup_steps)

# After warmup, use normal schedule (e.g. cosine)

Why warmup? At start, model is random, large LR causes instability

Gradient Clipping

The Problem:
Occasionally get very large gradients (exploding gradients)
gradient norm = 1000 (normal is ~1)
Update: weights += -0.001 × 1000 = weights - 1
Model diverges!

The Solution:
max_norm = 1.0
total_norm = sqrt(sum(grad**2 for all grads))

if total_norm > max_norm:
    scaling_factor = max_norm / total_norm
    for grad in all_grads:
        grad *= scaling_factor

# Now total norm is exactly max_norm

Mixed Precision Training

The Idea: Use 16-bit floats (FP16) instead of 32-bit (FP32).

BenefitImprovement
Memory2x reduction (28 TB → 14 TB)
Compute2-8x faster on modern GPUs
Bandwidth2x less data to move around
The Problem: Underflow
FP16 range: ~6e-5 to 65,000
FP32 range: ~1e-38 to 1e38

Small gradients in FP16:
  0.00001 in FP32 → 0 in FP16 (underflow!)
  Important updates lost!
The Solution: Loss Scaling
# Forward in FP16
loss = model(input)            # loss might be 0.01

# Scale loss before backward
scaled_loss = loss * 1000      # now 10

# Backward in FP16 (gradients also scaled by 1000)
scaled_loss.backward()

# Unscale gradients before optimizer step
for param in model.parameters():
    param.grad /= 1000

# Update in FP32 (keep master weights in FP32)
optimizer.step()
model.fp16_weights = model.fp32_weights.half()
Dynamic Loss Scaling:
scale = 2**16  # start high

while training:
    scaled_loss = loss * scale
    scaled_loss.backward()

    if has_inf_or_nan(gradients):
        scale /= 2              # reduce scale
        skip_this_step()
        continue
    else:
        if last_N_steps_successful:
            scale *= 2          # try increasing

    gradients /= scale
    optimizer.step()

Load Balancing in Pipeline Parallelism

The Problem:
GPU 0 (Embedding):     100ms  per micro-batch
GPU 1 (Layers 1-10):   500ms  per micro-batch  ← BOTTLENECK
GPU 2 (Layers 11-20):  300ms  per micro-batch
GPU 3 (Output):        100ms  per micro-batch

Overall throughput limited by slowest stage.

Solution: Uneven Layer Distribution
GPU 0: Embedding + Layers 1-3   (250ms)
GPU 1: Layers 4-8               (250ms)
GPU 2: Layers 9-15              (250ms)
GPU 3: Layers 16-20 + Output    (250ms)

Now balanced! Profile each layer first, then partition to equalize time.

Part 10: Common Pitfalls and Debugging

Synchronization Bugs

BAD: Only GPU 0 does something
if rank == 0:
    data = load_data()
# GPUs 1-3 don't have data! Will crash or hang

GOOD: Everyone loads data (or broadcast)
data = load_data()  # all GPUs execute

# OR broadcast from GPU 0
if rank == 0:
    data = load_data()
else:
    data = None
data = broadcast(data, src=0)

Deadlocks

BAD: Conditional communication
if rank == 0:
    send_to_rank_1(data)
if rank == 1:
    recv_from_rank_0()
# GPU 0 sends, but GPU 1 might not receive → deadlock

GOOD: Symmetric communication
if rank == 0:
    send_to_rank_1(data)
elif rank == 1:
    data = recv_from_rank_0()

Reproducibility

def set_seed(seed, rank):
    # Different seed per GPU for data sampling
    random.seed(seed + rank)
    np.random.seed(seed + rank)

    # Same seed for model initialization
    torch.manual_seed(seed)
    torch.cuda.manual_seed_all(seed)

Common bottlenecks:
  High "cuda wait"  → communication bottleneck
  High "memcpy"     → too much CPU-GPU transfer
  Unbalanced layers → pipeline imbalance

Memory Debugging (OOM)

torch.cuda.memory_summary()
peak = torch.cuda.max_memory_allocated()
torch.cuda.empty_cache()

Common causes:
  1. Activations too large      → use gradient checkpointing
  2. Too many micro-batches     → reduce pipeline depth
  3. Optimizer states           → use ZeRO
  4. Gradients accumulating     → call optimizer.zero_grad()

Summary: Decision Framework

ScenarioSolution
Model fits in GPU, want faster trainingData Parallelism
Model too big for one GPUTensor or Pipeline Parallelism
Model WAY too big (100B+)3D Parallelism (all three)
Limited GPU memoryZeRO or Gradient Checkpointing
Small number of GPUs (1-8)Data Parallel + DDP
Large GPU cluster (1000+)3D Parallelism + ZeRO
Consumer hardwareZeRO-Infinity (CPU/NVMe offload)
Research / experimentationData Parallel (simplest)
Production trainingWhatever works + checkpointing
Key takeaway: It's all about trade-offs — memory vs speed vs complexity.
